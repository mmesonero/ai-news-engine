# AI News Engine — 360° Code Audit

**Date:** 2026-06-25
**Method:** 9 parallel audit agents (one per dimension) → adversarial per-finding verification → synthesis. 64 agents, ~4.3M tokens.
**Result:** 52 issues confirmed, 3 false positives rejected. **0 critical**, 10 high, 17 medium, 25 low.
**Overall:** the codebase is well-architected and the documented defenses (prompt-injection sanitizing, enum validation, idempotency, archive-friendly retention) are largely real. The gaps are concentrated in four areas: **outbound-fetch SSRF**, **URL-scheme XSS on the public site**, **pipeline-level error resilience that contradicts the docs**, and **CI/supply-chain hardening**.

## High-severity findings (detail)

### H1 — SSRF + unvalidated image_url storage (`app/ingestion/image_extract.py`)
`_image_for()` GETs the feed-controlled article URL with `follow_redirects=True` and no scheme/host/IP check, extracts an `og:image`/`twitter:image` via regex, and stores it as `raw.image_url` with no `http(s)` validation. The stored URL is later rendered into `<img src>` on the public detail page (`static_site.py:532-533`), email `<img>` (`email_digest.py:113`), and Telegram `sendPhoto` (`telegram.py:97`).
**Impact:** server-side request forgery (fetch internal hosts) + arbitrary external/`data:` URL planted into public output. (Note: `javascript:`/`data:` in `<img src>` does **not** execute script in modern browsers, so the image sink is content-spoofing/SSRF, not stored XSS. SSRF blast radius is limited on GitHub-hosted runners — no cloud IMDS — but the primitive is real and would matter on self-hosted/cloud-VM runners.)
**Fix:** before fetching, require scheme ∈ {http,https}, resolve host and reject private/loopback/link-local/reserved IPs (`ipaddress`), disable or hard-cap redirects with per-hop re-validation; require `http(s)` + non-private host on the extracted og:image before storing.

### H2 — Stored XSS via `href` on public detail pages (`app/export/static_site.py:524`)
Source chips set `href="{_esc(s["url"])}"` from feed-controlled `RawContent.url`. `_esc` is `html.escape`, which blocks attribute breakout but **not** dangerous schemes — a `javascript:` URL is clickable and executes on the public GitHub-Pages site.
**Fix:** add a `safe_url()` helper applied to every `href`/`src`: allow only `http`/`https` (and the engine's own relative `n/<slug>.html`), drop everything else. `html.escape` is necessary but not sufficient in URL context.

### H3 — DOM XSS in admin dashboard (`app/static/index.html:349,377,412`)
`loadLinkedin`/`loadWeeklyTop`/`loadTrending` interpolate API `url` values straight into `href` inside `innerHTML` strings — titles go through `esc()`, URLs do not, and `esc()` wouldn't neutralize `javascript:` anyway.
**Fix:** add `safeHref(u)` returning `'#'` unless `u` starts with `http://`/`https://` (after trim, case-insensitive); use it for every href and still `esc()` the attribute.

### H4 — False player tags from prefix matching (`app/ai/players.py:35`)
Alias patterns compile `(?<![a-z0-9]) + alias` with a leading boundary but **no trailing boundary**, so any alias that is a prefix of a longer word matches: "a coherent argument"→Cohere, "grokking"→xAI, "applesauce"→Apple, "codexample"/"sorandor"→OpenAI. Directly violates the documented "NEVER produce false player tags" rule; mis-tags flow into LinkedIn hashtags, email/web chips.
**Fix:** add a trailing boundary `(?![a-z0-9])`. Aliases that intentionally end in space/punctuation (`'meta '`, `'gpt '`, `'dall-e'`) must be stripped first or special-cased so they don't lose their whole-word semantics.

### H5 — N+1 query explosion in export (`app/export/static_site.py:330-361`)
`_collect` runs the main query (limit up to 5000 for the full archive) then issues 2 sequential queries **per row** (cluster_id lookup + sources list). That's ~10,000 round-trips against Neon per publish. The main `stmt` already `outerjoin`s `ClusterItem` (line 313), making the per-row cid lookup redundant.
**Fix:** collect all rep ids, resolve cluster_ids in one query, fetch all `(cluster_id, source_name, url, raw_id)` rows with a single `cluster_id IN (...)`, group in Python.

### H6 — No lockfile, open `>=` ranges (`pyproject.toml`, all workflows)
Every dep has an open lower bound, no upper cap, and there is **no lockfile**. Each cron run does `pip install -e .` and resolves the latest satisfying version of every direct + transitive dep, in a job holding `OPENAI_API_KEY`, `DATABASE_URL`, `TELEGRAM_BOT_TOKEN` (and `PAGES_TOKEN` in the publish step). A compromised/typosquatted/regressed PyPI release lands silently into the secret-bearing run.
**Fix:** generate and commit a hashed lockfile (`uv lock` / `pip-compile --generate-hashes`); install with `--require-hashes` / `uv sync --frozen` instead of `pip install -e .`. Add Dependabot/Renovate to bump the lock deliberately.

### H7 — Whisper cost cap is per-source, not per-run (`app/ingestion/youtube.py:77`)
`whisper_calls_remaining = settings.whisper_max_per_run` is initialized **inside** `fetch()`, which runs once per source. With N YouTube sources the real ceiling is `N × whisper_max_per_run` (default 15 each), contradicting the docstring and `config.py` ("per pipeline run"). Bounded further by upstream gating, but still scales linearly with source count.
**Fix:** track the budget run-scoped — pass a shared mutable counter from `IngestionService` into each ingestor, or enforce a module-level counter in `fetch_transcript_via_whisper` reset once at pipeline start.

### H8 — No per-stage error handling in the daily pipeline (`app/pipeline/daily.py:36-101`)
~13 stages run sequentially with **no** try/except. The documented "one bad article never aborts a run" resilience only exists inside ingestion and per-item classify/enrich. The genuine uncaught abort points are `EmbeddingService.embed_pending` (reraises after 5 retries) and `DedupClusteringService.process_new` (no try/except on pgvector SQL/commits) and the holistic LLM merge — any of which propagates and **skips all delivery** (`send_new_stories`, `update_boosted_stories`, `build_and_send_breaking`).
**Fix:** wrap each stage in try/except that logs and continues (keep delivery reachable even if an upstream stage fails); accumulate errors and re-raise/exit non-zero at the end so the Actions run still goes red — but only after delivery is attempted.

### H9 — `feedparser.parse(url)` has no network timeout (`app/ingestion/rss.py:29`, `youtube.py:207`)
Passing a URL makes feedparser do the HTTP fetch itself via urllib with no timeout. A hung server blocks the worker until GitHub kills the job at `timeout-minutes: 30`; one dead feed consumes the whole budget and starves later sources/stages. (`feedparser.parse` accepts no timeout arg.)
**Fix:** fetch bytes via `httpx` with an explicit timeout and pass `feedparser.parse(content)`; add a per-source `asyncio.wait_for` watchdog in `ingestion_service._ingest_source` to also cover Html/article fetchers.

### H10 — LinkedIn translation bypasses the injection guard (`app/notify/linkedin_draft.py:42-71`)
Every other LLM call prepends `INJECTION_GUARD` and `sanitize.wrap()`s untrusted text. `_translate_to_es` does neither — it builds the prompt from raw `title`/`summary` (`proc.title_es or raw.title`, where `raw.title` is the untrusted ingested headline). An attacker controlling a headline can inject instructions; the output becomes a LinkedIn draft the operator is primed to copy-paste. This is the counterexample to `SECURITY.md:21`'s "en cada llamada LLM" claim. (Human-in-the-loop lowers blast radius but the operator is primed to publish.)
**Fix:** prepend `INJECTION_GUARD` to the system prompt and wrap each item's title/summary with `app.ai.sanitize.wrap()`/`neutralize()`, matching every other caller.

## Medium-severity findings (detail)

- **M1 / M2 — SSRF in `article_fetcher` and `HtmlIngestor`.** Same root cause as H1: feed-controlled URLs fetched with `follow_redirects=True`, no scheme/private-IP validation; `HtmlIngestor` additionally crawls every discovered href with no same-domain confinement. **Fix once, apply to all four fetchers** (article_fetcher, image_extract, html, youtube): a shared `safe_fetch` that validates scheme + blocks private/loopback/link-local/reserved IPs + re-validates redirect hops.
- **M3 — Representative reselection can pick a pruned member** (`enrichment_service.py:149`). The candidate query lacks `.where(RawContent.embedding_pruned.is_(False))` that `cluster_merger` uses elsewhere; a later run re-entering enrichment (e.g. the `title_es IS NULL` retranslate path) can promote a text-blanked row and enrich on an empty body. Self-heals next run via `repair_orphan_representatives`, but after the degraded result already published. **Fix:** add the pruned-exclusion filter, fall back to current rep.
- **M4 — Neon pool config** (`database.py:19-23`). Only `pool_pre_ping=True`; no `pool_recycle`, and asyncpg prepared-statement cache left on (breaks against the `-pooler`/PgBouncer endpoint). Conditional on using the pooled endpoint. **Fix:** `pool_recycle=300`; if pooled, `connect_args={"statement_cache_size": 0}` or use the direct Neon endpoint for the cron.
- **M5 — ivfflat recall** (`0001_initial.py:102`). Built `lists=100` on an empty table and no `ivfflat.probes` set (defaults to 1 → scans ~1% of vectors) → real duplicates can fall below threshold and be missed. **Fix:** `SET LOCAL ivfflat.probes` before `nearest_within`, or switch to HNSW; REINDEX once the corpus is populated.
- **M6 — Missing `fetched_at` index** (`0001_initial.py:47`). Hot filter in retention, dedup join, cluster merger, telegram, trending, weekly_top, stats, export. **Fix:** add an index (ideally composite/partial on `(published_at, fetched_at)` to serve the common OR-filter).
- **M7 — Naive YouTube `published_at`** (`youtube.py:219`). `datetime.fromtimestamp(mktime(tup))` with no tz vs the correct `rss.py:73` (`tz=timezone.utc`). Near-correct on UTC Actions runners, skewed on local runs. No Python TypeError (comparisons happen server-side in Postgres). **Fix:** `from datetime import timezone` + `tz=timezone.utc`.
- **M8 — Stale "deprecated columns" docs** (`ARCHITECTURE.md:151`, `CHANGELOG.md:85`). `linkedin_potential_score`, `linkedin_angles`, `ai_generated_insights` are documented "unused" but are read by `/weekly-top` and `/linkedin-ideas` (mounted, and `static/index.html` fetches `/linkedin-ideas`) and serialized via `ProcessedRead` (declared non-nullable). Dropping them would break live endpoints. Only `novelty_score` + `business_impact_score` are genuinely not consumed in business logic (see L-section). **Fix:** correct the docs; don't drop the three used columns without removing dependents first.
- **M9 / M10 / M12 — Actions hardening.** Mutable `@v4/@v5` tags (SHA-pin + Dependabot), no `permissions:` block (add `permissions: contents: read` / `{}`), and `persist-credentials:true` default leaving `GITHUB_TOKEN` in `.git/config` next to `PAGES_TOKEN` (set `persist-credentials: false`; move the PAT publish to its own minimal job).
- **M11 — No dependency vuln scanning** on the secret-bearing job that ingests untrusted media (yt-dlp) and parses untrusted RSS/HTML (lxml). **Fix:** lockfile + a `pip-audit` CI step that fails on known-vulnerable resolved versions. *(The original "yt-dlp < 2024.07 CVE" example was wrong — a `>=2024.10` floor excludes it; the real issue is the missing lock+audit, not the floor.)*
- **M13 — Unbounded response buffering** (`article_fetcher.py:64-70`, `image_extract.py:58`, `html.py:38,53`). `client.get()` then `.text`/`BeautifulSoup` reads the whole body before truncating; a multi-GB body OOMs the runner. The image_extract content-type check runs *after* the body is buffered. **Fix:** stream with a byte cap, or check `Content-Length`/`Content-Type` before reading.
- **M14 — OpenAI client timeout** (`openai_client.py:23-27`). No `timeout=` → SDK default 600s, longer than the 30-min job; `APITimeoutError` is retryable so it can recur across 5 tenacity attempts, and the SDK's own `max_retries=2` doubles retries. **Fix:** `AsyncOpenAI(timeout=httpx.Timeout(60.0), max_retries=0)`.
- **M15 — `theme` not enum-validated + unescaped** (`classification_service.py:50` → `static_site.py:419`). `importance_tier` is clamped to `{high,medium,low}` but `theme` is stored verbatim (only `.lower()`) and interpolated raw into `data-theme="..."`. A `theme` value containing a `"` would break out of the attribute (second-order stored XSS, needs the LLM to defeat the guard and emit a quote). **Fix:** validate `theme` against the enum (fallback `other`) **and** `_esc()` it at render. *(The companion claim about `tier`/`data-tier` is a false positive — tier is already enum-clamped.)*
- **M16 — Config validator masks misconfig** (`config.py:98`). `_blank_int_to_default` substitutes the default for any blank/whitespace value with no log, and there are no range checks anywhere (`telegram_min_score="999"` or `email_port="70000"` pass through). **Fix:** log when substituting a set-but-blank value; add `ge/le` bounds via `Field`.
- **M17 — `ENABLE_SCHEDULER=true` default** (`config.py:15`, `main.py:37`). Serving `app.main:app` (the README's documented `uvicorn` command) starts an in-process APScheduler that fires the full real pipeline (OpenAI spend + Telegram sends) at `0 6 * * *`. Cloud is safe only because CI invokes the module directly, not via uvicorn. **Fix:** default to `False` (opt-in), or document the live-fire risk next to the uvicorn command.

## Low-severity findings (25)

**Security / repo hygiene**
- **L1** Tracked junk file with a fullwidth-colon path `C：/Program Files/Git/app/site/index.html` (stale Spanish render, no secrets) — one-off local Git-Bash MSYS path-mangling quirk, not a CI defect. `git rm` it and commit.
- **L2** `.gitleaks.toml` allowlists whole `README.md`/`ARCHITECTURE.md` (and, unanchored, nested `*/README.md`) → permanent secret-scan blind spot in exactly the docs most likely to get a pasted key. Narrow to specific placeholder regexes.
- **L3** Admin `/admin/run-pipeline` + `/admin/run-retention` have no auth. Low because the cloud never serves the HTTP API (CI calls Python entrypoints directly) and CORS is localhost; still, gate behind a shared-secret dependency. *(retention "deletes" only embeddings + raw_text; the archive rows are kept by design.)*

**Correctness / DB**
- **L4** `cluster_for()` uses `scalar_one_or_none()` but `raw_content_id` isn't unique across clusters → latent `MultipleResultsFound` if the app invariant ever breaks. Add a UNIQUE constraint or `.limit(1).first()`.
- **L5** `update_boosted_stories` has a `LIMIT 30` with no `ORDER BY` → non-deterministic subset edited; a persistent >30/run backlog can starve some edits (cosmetic counter staleness). Add a stable `order_by`. *(`send_new_stories` is fine — it already orders.)*
- **L6** `DedupClusteringService.process_new` commits once per candidate with no try/rollback (many tiny Neon txns). Bounded by `async with SessionLocal()` auto-rollback + idempotency. Batch commits + per-item try/except.
- **L7** No partial index on `raw_content.embedding_pruned` (smaller win than it looks — the embed-pending scan is gated by the Embedding antijoin).
- **L8** `_find_entity_overlap_pairs` is O(n²) Python over all recent reps each of 8 merge rounds; cost is masked by the LLM round-trips but scales poorly. Invert `key_topics`→cluster ids and bucket.
- **L9** `reprocess` endpoint calls `EnrichmentService(session)._enrich_cluster` on the request session, pinning a pooled connection across multiple LLM calls and committing a session it doesn't own. Move to a background task with its own `SessionLocal`.

**Dead / partially-dead code**
- **L10** `novelty_score` + `business_impact_score` are LLM-computed every enrichment and serialized, but never consumed in business logic (only `viewer.html` dev-only sorts use them). Either drop them from `ENRICH_V1` (saves tokens) or document as live — pick one.

**Supply chain (lower tier)**
- **L11** `psycopg2-binary` as a prod dependency (Alembic path only; app runtime is asyncpg, so impact is narrow). Move to a dev extra / use source `psycopg2` or `psycopg3`.
- **L12** Workflows `pip install --upgrade pip` every run (unpinned toolchain). Pin or rely on the (SHA-pinned) setup-python's bundled pip.
- **L13** No `dependabot.yml` / CodeQL / server-side scan (gitleaks is local pre-commit only).

**Resilience (lower tier)**
- **L14** `image_extract.backfill_images` opens `httpx.AsyncClient()` with no client-level timeout (every real request sets per-request `timeout=10`, so no live hang — latent footgun + no overall loop budget). Set `timeout=10` on the client.
- **L15** Telegram `_send_one` does `int(data["result"]["message_id"])` with no guard; a non-standard `ok:true` response missing the field raises `KeyError` and (no per-story try/except) aborts the delivery loop. Use defensive access + wrap the loop body.
- **L16** `backfill_players` scans + rewrites all `ProcessedContent` each run, single transaction, no per-row isolation; raises only if `key_topics` holds a non-str. Limit to recent/NULL rows + batch + per-row try/except.

**LLM safety (lower tier)**
- **L17** No `max_tokens` on any completion (output bounded only by model cap). Real effect is incremental cost, not unbounded (JSON schema + clamped consumers discard padding). Add a sane `max_tokens`.
- **L18** Free-form classifier `category`/`tags` stored verbatim as `key_topics`/`rejected_reason` with no enum/shape validation. No crash and merges still need the LLM same-event judge, so dedup poisoning is overstated; residual is data-quality. Coerce/cap tags.
- **L19** `json_completion` returns `{}` on parse failure (and on valid-but-empty JSON, with no log) → the noise classifier fails *open* to a publishable "medium". Fail-closed (mark `is_noise`) on empty payload + emit a counter.

**Docs / config consistency**
- **L20** `ARCHITECTURE.md:274` says `TELEGRAM_MIN_SCORE` default 85; code is 65 and cloud never overrides it. (`README`/`.env.example` are correct.)
- **L21** `README.md:139` says `RETENTION_DAYS` "(14)"; code default is 30, only cloud sets 14. Align with `ARCHITECTURE.md:271`.
- **L22** `CHANGELOG.md:73` says web bakes 30 days; `RECENT_DAYS=90` and other docs say 90 (stale historical entry).
- **L23** `whisper_max_per_run` (and other env-driven int/float fields) omitted from the `field_validator` blank-coercion list → would crash startup if ever wired to a blank secret. Not currently in CI. Add to the list.
- **L24** `PUBLIC_SITE_BASE` is read from a repo var in `weekly_email.yml` but from the code default in `daily.yml` (which generates the detail pages + Telegram/LinkedIn links). A stray var would diverge link bases. Read it the same way in both. *(No host typo — every reference uses `mmesonero.github.io`.)*
- **L25** (rolled into M15) `theme` rendered unescaped — the low-severity attribute-injection variant; same fix.

## Rejected findings (verified false positives)

These were raised by finders and **disproven** during adversarial verification — kept for the record so they aren't re-raised:

1. **"Telegram bot token leaks into Actions logs via uncaught httpx error"** — empirically false. Tested with the project's httpx: `TOKEN in traceback.format_exc()` is `False`; the token lives on `e.request.url`, which the default traceback formatter does not render, and the source line shows the `{settings.telegram_bot_token}` f-string literal, not the value. *(Residual real-but-low: the missing try/except in `_send_to_telegram` is an availability gap — a network blip there aborts the run after Telegram delivery, skipping retention + publish.)*
2. **"Blocking smtplib inside the async event loop in `build_and_send_weekly`"** — false call chain. `build_and_send_weekly` never calls `send_weekly_digest`/`_send_via_smtp`; SMTP runs only from `email_digest.main()` as a separate single-shot `asyncio.run` process with no concurrent tasks.
3. **"`RETENTION_DAYS == DEDUP_LOOKBACK_DAYS` truncates the dedup window"** — mathematically wrong. Retention always runs with an earlier clock than the next dedup, so the pruned range (`fetched_at < T0−14d`) is strictly older than the dedup-needed range (`fetched_at ≥ T1−14d`) by the inter-run gap. Equality is sufficient; `>=` (already documented) is the correct invariant.

## Recommended fix order

1. **One shared SSRF/URL guard** → fixes H1, M1, M2 (+ image-url storage) and feeds the XSS fix. A single `safe_fetch()` + `safe_url()` module.
2. **URL-scheme allowlist for `href`/`src`** → H2, H3, M15 (public + admin XSS).
3. **Pipeline resilience** → H8 (per-stage try/except) + H9 (feed timeout) + M13/M14 (fetch size cap, OpenAI timeout) + L15 (telegram guard). These are the "the run silently does nothing / hangs" class.
4. **Cost controls** → H7 (per-run whisper cap) + L17 (max_tokens).
5. **LLM injection parity** → H10 (LinkedIn translation guard) + M15 (theme enum).
6. **Supply chain** → H6 (lockfile) + M9/M10/M11/M12 + L11/L12/L13 (one CI hardening pass).
7. **Quick wins** → H4 (player regex boundary), M7 (tz), H5/M5/M6 (export N+1 + vector recall + index), M8 + L20–L24 (docs), L1 (`git rm` junk file).
